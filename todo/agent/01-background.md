## 1. 项目背景与目标

### 1.1 项目定位

CAT 是一个自托管的 Computer-Assisted Translation Web 应用。Agent 系统的核心目标是让若干 AI agent 在系统中 **扮演用户角色**，深度参与翻译的全链路：

- **主链路**: 条目入库 → 上下文发现 → 翻译 → 审校 → 发布
- **旁支链路**: 术语库维护、翻译记忆库维护、质量分析报告
- **治理链路**: 翻译任务编排与排布、参与翻译工作的企业员工的工作安排建议、项目成员权限管理辅助、项目元数据治理
- **标准制定链路**: 通过规范板 (NormsBoard, §3.26) 协作制定和维护翻译标准——人类和 Agent 均可提出规范条目，经审批后成为全项目黄金标准

Agent 可以被特定事件触发（如新文件导入），也可以通过人工调度触发，或者通过自身内存中的计划自行触发。**人类管理员可通过各种链路（对话、Issue/PR 指派、规范板、邮件、API）向 Agent 发送命令**，让 Agent 帮助管理翻译链路的全部内容。

### 1.2 核心架构原则

> 本节将贯穿全文档的核心设计理念提取为显式原则，供各子系统设计参照。

**原则 1 — Agent 即用户**: Agent 在本系统中不是"后台工具"而是一等公民用户。它们拥有用户身份、受 ReBAC 权限管控、通过同一套 operations 层执行业务操作、在 Issue 系统上领取和完成任务——和人类用户使用完全相同的业务接口。任何人类能做的操作，Agent 也应能做（受权限约束）；任何 Agent 能做的操作，人类也能以相同方式干预。

**原则 2 — Agent/Team 同构 (TaskExecutor Isomorphism)**: 单个 Agent 和 Agent Team 在**任务接受与结果输出**层面是同构的。两者都能：接受相同来源的任务（Issue、事件、手动触发）；产出相同形态的结果（ChangeSet、Issue/PR 状态更新、通知）；被相同的调度机制（SchedulerService）触发和监控。这意味着任务既可以分配给独立 Agent，也可以分配给包含多个成员的 Team——区别在于协作质量和效率，而非能力边界。所有子系统在引用"Agent"完成某项功能时，该功能同样可由"Team"完成——包括但不限于 Review（§3.27.10）。该原则的具体接口设计见 §3.18。

**原则 3 — 全链路可追溯**: 从任务创建到最终完成，每一步都可审计追溯。Issue、Agent Session、ChangeSet、Events、Patches 构成一条完整的追溯链。任务的分解（Issue 拆分 §3.9.3.2）、流转、**委派** (§3.9.3.1) 在任何节点都可以回溯到源头。委派链不论深度几何，变更集始终归属原始任务；拆分后的子 Issue 支持聚合追溯（统一视图 + 逐 Issue 视图）。该原则的具体追溯模型见 §3.8。

**原则 4 — 经验可累积**: Agent 不是无状态的执行器。通过记忆系统（§3.13），Agent 能够跨会话积累翻译经验、领域知识和风格偏好，从重复劳动中学习并持续改进。记忆在 Agent、Team、Project 三个层级上共享和继承，使得整个系统具备组织级的知识沉淀能力。经验累积不意味着知识僵化——知识健康监控 (§3.28) 确保高权重知识在出错时能被及时发现和修正。

**原则 5 — 步骤级可观测**: Agent 的每一步推理、工具调用和决策都是独立可寻址的执行节点。系统在保留 LLM 完全决策权的前提下，通过通用 DAG 执行模型（§3.6）实现单步粒度的调试、重放、暂停和回退能力。重放 (replay) 使用录制的 LLM 历史响应而非重新调用，保证确定性 (§3.6.6)。这一原则确保从单 Agent 到多 Team 的全层级系统在出错时可以精确定位问题所在。

**原则 6 — 全链路参与**: 这套系统是**全链路**的——Agent/Team 的职责不仅限于翻译和审校等典型翻译场景。Agent 将长期参与：术语库的持续维护和治理、翻译记忆库的清洗与优化、翻译任务的排布与调度建议、对参与翻译工作的企业员工的工作安排建议、项目本身的治理（如成员权限管控建议、项目元数据维护）。人类管理员可通过对话、Issue/PR、邮件、API 等各种链路向 Agent 发送管理命令。**安全含义**: 全链路参与极大扩展了 Agent 的攻击面——提示词注入攻击可能诱导 Agent 尝试提权、修改关键权限配置甚至破坏整个项目。安全强化模型 (§3.25) 对此提供系统性防御。

**原则 7 — 人类内容至高性**: 在 Agent 时代，**人类工作的价值**是全平台的一等产出物。具体而言：人类的手动审核结果、人类介入批准的手动翻译、人类自行插入的术语、人类确认的错误翻译标记、人类在元素或翻译下的评论讨论——这些都是系统的**黄金标准 (Golden Standard)**。Agent 必须有意识地充分利用这些人类信号，而非"用 AI 内容生成 AI 内容"导致系统退化为通用机器翻译系统。架构层面的保障包括：人类内容在记忆系统中享有基于角色和信号类型的分级权重 (§3.13.9)；黄金标准修改时通过级联失效机制辐射到全系统 (§3.13.9)；Agent 的提示词中显式注入人类信号优先的指令 (§3.2)。黄金标准的权威性根据产出者权限角色和信号类型动态分级——项目译员的标准高于普通成员，拒绝原因高于任何翻译历史 (§3.13.4)。

**原则 8 — 双核交互 (Issue/PR 系统 + 规范板)**: 人类与 Agent 之间的核心交互分为两个并行维度——**Issue/PR 系统**用于任务派发和进度追踪，**规范板 (NormsBoard, §3.26)** 用于制定和维护翻译标准。Issue/PR 系统回答"做什么(What)"和"谁来做(Who)"；规范板回答"怎么做(How)"和"做到什么标准(Standard)"。两者共同构成完整的人类—Agent 协作框架。规范板的内容对人类和 Agent 都可读，既是 Agent 的长期记忆来源，也是项目的公共知识资产。

**原则 9 — 任务级验收闭环**: 一个健康的 agent 任务包含**规划、目标、规范和验收标准**四要素。Issue/PR 系统和规范板分别覆盖了前三者（Issue/PR 系统派发任务和目标，规范板制定规范），但**验收标准**——即"如何程序化地判定任务足量且高质量地完成"——不能依赖 LLM 自我评价（幻觉风险）。系统通过任务验收闭环子系统 (§3.27) 为每类任务动态注入**纯程序化** QA-checker 和结构化验收矩阵，使任务完成判定**可量化、可审计、不依赖 LLM 自评**。验收关卡**仅且只能**执行确定性检查（完整度、术语一致性、字符串匹配规范合规等），**不包含任何 Agent/LLM 参与的概率性判断**；概率性质量评审（如翻译自然度、风格适配度）由 Team 群体智能承担——Team 可包含任意配置的 Reviewer Agent 在会话过程中动态评估质量并反馈给其他成员进行迭代 (§3.27.10)，此模式通过 Pipeline with Feedback (§3.9 模式 4) 配置实现，并受通用 Team 会话终止机制 (§3.9.1) 保障不会无限循环。验收关卡与质量评审两者职责互补但**严格分离**: AcceptanceGate 是纯程序化守门人 (§3.27.9)，Reviewer Agent/Team 是 Team 内部的群体智能组件。

**原则 10 — 知识自愈性**: 高权重知识（黄金标准记忆、规范板规则）的权威性不是永恒的。如果错误的黄金标准被反复采用，系统的知识库会僵化（马太效应）。系统通过知识健康监控 (§3.28) 追踪每条高权重知识的**派生谱系**——当派生内容被频繁修改或放弃时，源知识的健康度自动下降，触发预警和动态降权。这确保了知识体系的自我修正能力，而非盲目信任历史权威。

**原则 11 — LLM 主权不可侵犯**: 系统的所有流程编排、通信拓扑、验收机制和任务分配**不得**以架构手段剥夺 LLM 的自主决策空间。具体约束：(1) DAG 执行模型是**调试基础设施**而非**决策管道**——系统不预设"这一步应调翻译工具"或"下一步必须审校"，LLM 在 ReasoningNode 中完全自主决定策略和行动；(2) Team 内邮件通信拓扑维持**严格 DAG 约束** (§3.10) 以保证通信可预测性和安全隔离——但 LLM 在需要跨领域协作时可通过 `delegate_task` (§3.9.3.1) 发起独立于邮件系统的委派通信，委派通信采用 TaskExecutor 统一接口的直接通道，不受 DAG 拓扑限制；(3) 验收系统和决策节点中的所有程序化判断（finish 检测、预算检查、步骤上限）是**边界守护**而非**行为指令**——它们划定安全和资源边界，但不指导 LLM "应该做什么"；(4) 委派 (§3.9.3.1)、Issue 拆分 (§3.9.3.2) 和动态组队 (§3.9.2) 是 LLM 发起的自主行为，系统提供机制但不强制特定编排模式。**违反此原则的设计信号**：任何子系统中出现"Agent 必须先做 X 再做 Y"的硬编码顺序约束（应改为 Skill 建议或 Issue/PR 系统任务级依赖）；任何通信路径中出现绕过权限检查的绝对封锁（应通过 SecurityGuard 权限审计实现访问控制，而非拓扑硬编码）。

**原则 12 — Agent/Team 单例性** _(v0.17 新增)_: 每个 Agent 实例和 Team 实例在系统中是**单例**的——同一时刻只有**一个活跃 Session**。任务并发不是 Agent 内部的并发，而是 Issue/PR 系统层面的编排职责：人类或调度器将同质任务聚合为单个 Issue（例如"翻译第 3 章 200 条"而非 200 个单条 Issue），Agent 按顺序逐个处理 Issue 条目。**设计动机**：(1) **记忆一致性**——若同一 Agent 并行执行多个 Session，其 MemoryStore (§3.13) 面临并发读写冲突（多个 Session 同时 create_memory、search_memory），goldenWeight 衰减和 healthScore 计算的原子性难以保证；(2) **状态简化**——单例 Agent 的 Scratchpad、黑板、DAG 执行状态均无并发竞争，大幅简化运行时实现；(3) **可观测性清晰**——Agent Timeline (§3.20) 是严格的线性时间序列，无需处理并行 Session 交织的复杂可视化。**任务吞吐量**通过 Team 水平扩展（多个同质 Agent 组成 Fan-out Team）和 Issue batchSize (§3.7.5) 聚合来保障，而非单 Agent 并发。

**原则 13 — 版本控制即平台基座 (VCS as Platform Substrate)** _(v0.23 新增)_: EntityVCS (§3.14) 不是仅在高安全场景下启用的可选组件，而是 CAT 平台的**核心基础设施**。以 **Project** 为中心，与其相关的一切业务实体都纳入版本控制——包括但不限于：翻译条目 (translation)、可翻译元素 (element)、文档与文档树 (document/document_tree)、元素下的评论 (comment)、项目设置 (project_settings)、项目成员 (project_member)、项目基本属性字段 (project_attributes)。**双重定位**：(1) **Agent 安全兜底**——VCS 是 Agent 系统从玩具走向生产的分水岭。无版本控制的 Agent 系统意味着任何 Agent 错误都可能不可逆地污染数据，这在企业翻译场景中不可接受。ChangeSet 审计 + 分支隔离 + 快照回归三级机制确保所有 Agent 操作可追溯、可审计、可回滚；(2) **TMS 平台基本能力**——作为翻译管理系统，CAT 应当提供对翻译资产的完整版本管理能力，这与 Agent 无关——即使没有 Agent 系统，人类用户同样需要查看变更历史、对比版本差异、回滚错误修改、在隔离分支中进行大规模变更。VCS 是 TMS 的一等公民功能，而非 Agent 的附属安全层。**架构含义**：所有业务实体的写操作（无论来自人类还是 Agent）均经过 VCS 层路由——Trust Mode 下直写 + 不记录、Audit Mode 下直写 + ChangeSet 审计、Isolation Mode 下写入工作分支 + 审核合并。VCS 层对上层调用者透明（由 Domain 层中间件拦截），各实体类型拥有专属的 diff 算法和 merge 策略 (§3.14.8)。

### 1.3 系统必要性与渐进式复杂度

#### 1.3.1 为什么需要 Agent 系统

单次 LLM 调用（将元素上下文、术语匹配、TM 匹配打包发送，返回一条翻译）可以覆盖 **点状翻译** 场景，但无法覆盖翻译 **全链路** 中的以下需求：

| 需求维度            | 单次 LLM 调用                    | Agent 系统                                                    |
| ------------------- | -------------------------------- | ------------------------------------------------------------- |
| **跨段一致性**      | 无法保证——每次调用独立           | 记忆系统 + Scratchpad 维护翻译决策一致性                      |
| **质量闭环**        | 无审校→反馈→修正循环             | Review TaskExecutor + Pipeline with Feedback 模式实现持续改进 |
| **术语治理**        | 依赖预置术语表，无法在翻译中演进 | TermManager Agent 持续维护术语库，翻译中发现的新术语即时入库  |
| **并行与协作**      | 无——串行逐条处理                 | 多 Agent 并行翻译，Coordinator 分配任务，Fan-out/Fan-in 聚合  |
| **人类监督**        | 无中间检查点                     | ChangeSet 审核、Issue/PR 系统、实时暂停/恢复/回溯               |
| **经验积累**        | 每次从零开始                     | 跨会话记忆使 Agent 越用越好                                   |
| **错误恢复**        | 无——失败即丢失                   | 时间线回溯、全实体 VCS 分支回滚、快照回归 (原则 13)           |
| **长文档 / 大项目** | 受上下文窗口限制                 | 任务分解 (DAG) + 多会话接力完成                               |
| **可审计性**        | 无追溯链                         | triggerContext → Session → Run → Event → ChangeSet 完整追溯   |
| **成本可控**        | 无成本感知                       | 多层级预算控制、自动降级、成本归因追溯 (§3.22)                |
| **全链路治理**      | 仅翻译单点                       | 术语维护、TM 治理、任务编排、权限管理全链路覆盖 (原则 6)      |
| **标准制定**        | 无                               | 规范板协作制定标准 + 自动注入 Agent 提示词 (原则 8, §3.26)    |
| **验收闭环**        | 无（LLM 自评不可靠）             | 程序化 QA-checker + 结构化验收矩阵 (原则 9, §3.27)            |
| **知识自愈**        | 无                               | 派生谱系追踪 + 健康度动态调整 (原则 10, §3.28)                |

核心结论：Agent 系统的价值不在于"翻译单条更好"，而在于 **编排、协作、质量闭环、知识沉淀、标准制定、验收保障和全链路治理**。对于只需要翻译单条的场景，直接调用 LLM 即可——这对应下文"复杂度 Level 0"。

#### 1.3.2 五级功能裁剪模型

系统按 **渐进式复杂度 (Progressive Complexity)** 设计，每个功能模块均可独立开关。组织或项目可根据实际需求选择启用级别：

| Level | 名称              | 启用的模块                                           | 典型场景                     |
| ----- | ----------------- | ---------------------------------------------------- | ---------------------------- |
| 0     | **Direct LLM**    | 无 Agent——直接经由 Operations 层调用 LLM 翻译        | 快速翻译、API 集成           |
| 1     | **Single Agent**  | AgentRuntime + Tools + Scratchpad + EntityVCS (Audit Mode) | 个人翻译助手、小文档翻译     |
| 2     | **Agent + 治理**  | Level 1 + 记忆 + 规范板 + 验收闭环 + EntityVCS (Isolation Mode 可选) | 需要审核和版本控制的正式项目 |
| 3     | **多 Agent 协作** | Level 2 + Team + Mail + Issue/PR DAG + 调度 + 知识健康 | 多语言大项目、翻译团队协作   |
| 4     | **全功能**        | Level 3 + 高级特性 (MCP, 自适应, 成本控制)           | 企业级翻译平台               |

- **✅ Decision D26: 功能裁剪实现策略** → 组合式模块注册 (B): 每个子系统以独立模块形式存在，启动时根据 Level 配置一次性组装运行时依赖图；未启用的模块不加载、不占用资源、不存在于运行时中——取代在代码中散落 if-else feature flag 检查的方式。

**组合式模块注册机制**:

```
┌──────────────────────────────────────────────────┐
│ Level 配置 (project | organization 级)            │
│   level: 2                                        │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│ ModuleComposer (应用启动时一次性组装)              │
│                                                    │
│ Level 0: DirectLLMModule                          │
│ Level 1: + AgentRuntimeModule, ToolRegistryModule │
│ Level 2: + ChangeSetModule, MemoryModule,         │
│            NormsBoardModule, AcceptanceGateModule  │
│ Level 3: + TeamModule, MailModule, SchedulerModule│
│            KnowledgeHealthModule                   │
│ Level 4: + MCPBridgeModule, CostControlModule     │
└──────────────────────────────────────────────────┘
```

**核心实现模式**:

| 模式                    | 说明                                                                                           | 示例                                                                            |
| ----------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Null Object**         | 低 Level 下子系统以空实现替代，接口不变但操作为空                                              | Level 0-1 下 `MemoryStore` 为 `NoopMemoryStore`（忽略写入，检索返回空）         |
| **Strategy Swap**       | 同一接口不同 Level 下绑定不同策略实现                                                          | Level 1 下 `VCSService` 为 `PassthroughVCS`（直writes DB）；Level 2+ 为真实 VCS |
| **Module Registration** | 模块在启动时注册中间件、事件监听器和服务实例；未注册模块的钩子不存在于管线中                   | TeamModule 注册 Issue/PR 事件监听和邮件路由；Level 1 下这些监听器不存在           |
| **Interface Stability** | 消费方始终调用相同接口 (`VCSService.write()`, `MemoryStore.search()`)，运行时实现由 Level 决定 | Agent 代码无需关心当前 Level                                                    |

**优势**:

- **零运行时条件分支**: Level 检查仅在启动时发生一次，执行路径中无 if-else
- **模块隔离**: 未加载的模块不占内存、不注册事件监听、不出现在中间件管线中
- **可测试性**: 每种 Level 组合作为独立 DI 配置，可在集成测试中完整覆盖
- **增量开发**: 新模块仅需实现接口 + 注册入口，不影响已有代码路径

**设计约束**: 高级别功能的关闭不应影响低级别功能的正确性。Level 1 的单 Agent 必须在 Level 3/4 的模块全部以 Null Object 替代时仍能正确工作。

### 1.4 与编码 Agent 系统的区别

参考了 Claude Code 等编码 agent 系统的架构（见 `.cc/`），但本系统有根本性差异：

| 维度       | 编码 Agent (Claude Code) | CAT 翻译 Agent                                  |
| ---------- | ------------------------ | ----------------------------------------------- |
| 交互模式   | 单用户对话式             | 多 agent 协作 + 人工在环                        |
| 工具性质   | 文件/终端操作            | 业务操作（翻译/术语/记忆库）                    |
| 上下文来源 | 代码仓库                 | 翻译项目 + 术语库 + TM                          |
| 执行时长   | 分钟级                   | 小时/天级（持续运行）                           |
| 输出形态   | 代码变更                 | 翻译条目 + 审校意见                             |
| 协作模式   | 单 agent + delegation    | Agent Team + Issue/PR + 邮件 + 委派               |
| 权限模型   | 隐式（用户权限）         | 显式 ReBAC + ChangeSet 审计                     |
| 任务来源   | 用户对话                 | Issue/PR + 事件 + 定时 + 对话                     |
| 知识持久化 | CLAUDE.md / rules        | 多层级记忆系统 + 规范板 + 全实体版本控制 (原则 13) |
| 复杂度裁剪 | 无（单一模式）           | 五级渐进式功能裁剪 (§1.3.2)                     |
| 执行模型   | while 循环               | 通用 DAG (步骤级可调试)                         |
| 安全模型   | 用户信任边界内           | 多层安全隔离 + 提示词注入防御 (§3.25)           |
| 标准制定   | CLAUDE.md 人工维护       | 规范板人类—Agent 协作制定 (§3.26)               |
| 验收判定   | 单元测试/集成测试        | 程序化 QA-checker + 验收矩阵 (§3.27)            |
| 质量评审   | 人工 review              | Pipeline with Feedback + Team 群体智能 (§3.9.1) |
| 知识健康   | 人工维护 rules           | 派生谱系追踪 + 自动降权预警 (§3.28)             |
