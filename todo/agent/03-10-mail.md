### 3.10 Agent 通信系统 (协议驱动)

> Agent 通信系统是 **Team 内部的 Agent 间结构化通信系统**。所有 Agent 间消息均为协议消息，"邮件"是面向用户和 LLM 的具象化比喻 (D53)。通信系统与 Issue/PR（任务管理）和 Message/站内信系统（系统通知 + 人类在环通知）在职责上严格区分。

**三个通信系统的边界**:

| 系统                      | 参与方            | 职责                                                | 实现基础                              |
| ------------------------- | ----------------- | --------------------------------------------------- | ------------------------------------- |
| **Agent 通信 (协议驱动)** | Agent ↔ Agent     | Team 内部任务协作（审校意见、术语讨论、委派回传等） | `agent_mail` 表 (协议消息), Team 内部 |
| **Issue/PR**              | Agent/人类 ↔ 系统 | 任务分配与状态流转                                  | `issue`/`pull_request` 表             |
| **站内信/通知**           | 系统 → 人类       | 系统通知、人类在环交互请求                          | `packages/message`                    |

**数据模型** _(v0.22 协议驱动重设计)_:

```
agent_mail (D2 已决策: 独立表; 表名保留——"邮件"作为 UI 比喻)
  ├── id, teamId, fromAgentId, toAgentId
  ├── subject: string                        -- 人类可读摘要 (可从 protocol 自动生成)
  ├── body (markdown)                        -- 人类/LLM 可读描述 (从 protocol 自动渲染或手动提供)
  ├── replyToMailId (线程化)
  ├── priority: NORMAL | URGENT
  ├── status: UNREAD | READ | ARCHIVED
  ├── attachments (jsonb: 引用资源列表)
  ├── protocol (jsonb, NOT NULL)             -- 结构化协议载荷 (D53: 必填)
  │     ├── type: ProtocolType
  │     ├── requestId: string?
  │     ├── payload: Record<string, unknown>
  │     └── expiresAt?: Date
  └── createdAt
```

- **✅ Decision D38: Team 通信拓扑约束模型** → Team 内邮件维持严格 DAG 约束 (A); 委派通信 (`delegate_task`) 独立于邮件系统，采用 TaskExecutor 统一接口的直接通道，会话结束后自动回传结果给委托者，不受 DAG 拓扑限制。跨领域协作需求通过委派机制解决，而非放松邮件拓扑。

**邮件通信安全规则**:

```
send_mail 安全检查 (SecurityGuard):
  1. 同 Team 内: 允许 (无论拓扑)
  2. 跨 Team 同 Project: 需要 cross_team_mail 权限
  3. 跨 Project: 禁止
  4. 安全级别检查: sender.securityLevel 与 receiver.securityLevel 差距不得超过 1 级
  5. 委派链内: delegationChainId 相同的 Agent 间自动授权通信
```

> **v0.15 变更**: D38 最终决定 Team 内邮件维持严格 DAG 约束（D7 原设计不变）。委派场景下的跨 Team 通信需求通过 `delegate_task` (§3.9.3.1) 的独立通信通道解决——委派通信采用 TaskExecutor 统一接口直接回传模式，不经过邮件系统。委派链内自动授权规则仍然适用。

**邮件触发机制**:

1. Agent 调用 `send_mail` 工具主动发送
2. 收到邮件的 agent 在 PreCheck Node（§3.6.4）中被注入邮件内容
3. `URGENT` 邮件在当前 Tool Node 执行完成后的下一轮 PreCheck 中即被处理
4. 普通邮件根据 D29 PreCheck 注入策略，可能被分类为 ROUTINE 级别写入 precheck_notes

- **✅ Decision D16: 邮件投递保障** → 混合模式 (D): `send_mail` 工具为主要发送方式，TeamCoordinator 监控通信模式并在检测到缺失时提醒 Agent 或自动注入上下文给下游。

**跨 Team 通信**: 具有 `cross_team_mail` 权限的 Agent 可跨 Team 发送邮件。委派链内的 Agent 因共享 `delegationChainId` 自动获得相互通信权限。

**与现有 message 包的关系**: `packages/message` 处理 **人 ↔ 系统** 的通知（站内信、邮件、webhook）。Agent 邮件系统处理 **agent ↔ agent** 的 Team 内部通信。当 agent 需要通知人类时，通过 `message` 包路由。

**与死锁检测的关系** _(v0.20)_: 邮件通信本身因 Team 内 DAG 约束 (D38) 不会产生维度内循环。但 URGENT 邮件的等待回复状态会被纳入 WaitGraphService 统一等待图 (§3.9.4)——当 Agent A 发送 URGENT 邮件给 B 并在后续推理中依赖 B 的回复时，此等待关系可能与 Issue 依赖或委派 WAIT 共同形成跨子系统环路。WaitGraphService 的离线检测覆盖此场景。

#### 3.10.2 统一通信协议 _(v0.22 重设计)_

> **D53 最终决策**: 引入统一通信协议覆盖一切 Agent 间通信用例。"邮件"只是这套结构化协议的具象化比喻，不再存在独立于协议之外的纯文本邮件通道。

**设计哲学**: 在 v0.21 中，§3.10.2 将结构化协议定位为 agent_mail 上的可选附加层。v0.22 反转这一关系——**协议即通信本体**，人类可读的文本描述是协议载荷的派生物（类似 HTTP 协议与浏览器渲染页面的关系）。

**为什么统一**: Agent 间的每次通信从根本上都有可形式化的语义（审校意见有 verdict、委派结果有 status code、术语讨论有候选列表）。v0.21 的"纯文本邮件 + 可选协议层"设计导致两个问题：(1) 非协议消息无法被机器解析，系统只能猜测意图；(2) 协议消息和纯文本消息走不同的处理路径，增加边际复杂度。统一后所有消息都是协议消息，系统始终拥有结构化语义。

**数据模型演进**: 参见上方 §3.10 数据模型——`protocol` 从 nullable 变为 `NOT NULL`，所有消息均为协议消息。

**ProtocolType 枚举**:

| ProtocolType        | 语义          | payload 关键字段                                                                    | 典型场景                 |
| ------------------- | ------------- | ----------------------------------------------------------------------------------- | ------------------------ |
| `general`           | 通用自由消息  | `{ text: string }`                                                                  | 非结构化沟通（向后兼容） |
| `review_verdict`    | 审校判定      | `{ verdict: "approve"\|"request_changes"\|"reject", comments: string[] }`           | Reviewer → Translator    |
| `delegation_result` | 委派结果回传  | `{ status: "completed"\|"failed"\|"partial", result: unknown, tokenUsage: number }` | 子任务 → 委派者          |
| `term_vote`         | 术语提议投票  | `{ termId: string, candidates: string[], vote?: string, deadline?: Date }`          | TermManager 发起投票     |
| `status_sync`       | 心跳/状态同步 | `{ state: string, progress?: number, eta?: Date }`                                  | 长任务进度汇报           |
| `task_assignment`   | 任务分配通知  | `{ cardId: string, role: string, instructions?: string }`                           | Coordinator → Agent      |
| `feedback`          | 质量反馈      | `{ targetSegmentIds: string[], issues: FeedbackIssue[] }`                           | Reviewer → Translator    |
| `escalation`        | 升级通知      | `{ reason: string, severity: "warning"\|"critical", context: unknown }`             | Agent → Coordinator      |

> **可扩展性**: ProtocolType 采用字符串枚举而非 sealed union，允许 ModuleComposer (§3.12) 注册的模块和 Hook (§3.29) 引入自定义协议类型，无需修改核心枚举。自定义类型建议使用命名空间前缀 (如 `plugin:my_protocol`)。

**body 自动渲染**: `send_mail` 工具接受 protocol 对象后，MailService 根据 ProtocolType 自动渲染 body (markdown)。Agent 也可在 `send_mail` 中显式提供 body 覆盖自动渲染结果。这确保：

1. **LLM 可读** (Principle 11): Agent 在 PreCheck 中收到邮件时，始终有人类可读的 markdown body
2. **机器可解析**: 系统在处理邮件时直接解析 `protocol.payload`，不依赖自然语言理解
3. **不强制 Agent 构造语义结构**: 对于自由沟通场景，Agent 只需 `send_mail(to, subject, body)` — 工具自动填充 `protocol: { type: "general", payload: { text: body } }`

**send_mail 工具签名更新**:

```
send_mail(to, subject, body?, protocol?) → MailResult
  如果提供 protocol:
    → 使用 protocol 作为结构化载荷
    → body 未提供时自动从 protocol 渲染
  如果仅提供 body:
    → 自动构造 protocol: { type: "general", payload: { text: body } }
  至少需要 body 或 protocol 之一
```

**接收方处理**: Agent 通过 PreCheck (§3.6.4) 收到邮件时，同时获得 body (在消息文本中) 和 protocol (在工具元数据中)。Agent 推理过程中 LLM 读取的是 body；系统级逻辑（如 SchedulerService 处理 `delegation_result`）直接解析 protocol.payload。

- **✅ Decision D53: Agent 间通信结构化协议** → 统一通信协议 (D——超越原始 A/B/C 选项)。结构化协议成为邮件系统本体，所有 Agent 间通信均为协议消息。`protocol` 字段从可选变为必填; `body` 从主体变为 protocol 的人类可读派生; 纯文本通信通过 `general` 类型向后兼容。此设计消除了结构化与非结构化双通道的复杂性，使系统始终拥有机器可解析的通信语义。
